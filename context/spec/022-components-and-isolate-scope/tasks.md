## Notes for the Implementation Agent

- **No new `EXCEPTION_HANDLER_CAUSES` entry.** The `'$compile'` token covers every new error site (binding, multiple-isolate, missing-require, invalid-component). The tuple stays at 10.
- **`Scope.$new(true)` already exists** — isolate-scope CREATION infra is in `@core`; this spec only wires the compiler to use it. Do not reimplement isolate scoping in `@compiler`.
- **The `later` instantiation ordering is fixed:** construct → stash into `$$ngControllers` → wire bindings → resolve `require` → `$onInit` → pre-link → child link → post-link → `$postLink`. A shared-spy ordering test pins it.
- **Both per-element link sites** in `compile.ts` (transcluding-host path + non-transcluding path) need the isolate-scope + binding + lifecycle + require wiring — same dual-site pattern spec 020 Slice 4 dealt with.
- **`$onChanges` fires for `<` and `@` bindings only** — not `=` or `&`. Batched per-digest via `scope.$$postDigestQueue`, plus one initial synchronous call at link time.
- **Preserve spec 017/018/020 behaviour exactly** — `scope: false | true`, the transclusion outer-scope rule, the spec-020 controller seam. Their regression suites are the guard.

## Slice 1 Implementation Notes (spec 022)

- **`IsolateScopeNotSupportedError` is RETIRED as a deprecated no-op.** The throw site in `normalizeDirective` has been removed. The class itself stays exported from `@compiler/index` and the root barrel for a one-release deprecation grace period so consumers catching it via `instanceof IsolateScopeNotSupportedError` keep compiling without a sudden `ReferenceError`. JSDoc carries a `@deprecated` tag; lint rule `@typescript-eslint/no-deprecated` is active and the inline disables at the two re-export sites are intentional (with `-- reason` justifications). A future spec may remove the class outright.
- **`scope` flag stays `boolean` on the normalized `Directive`.** Option (b) from the brief — the `scope: true | false` flag continues to signal "needs a non-default scope," and `isolateBindings != null` is the runtime "isolate vs. child" discriminator. No public-API breakage for existing `scope: false | true` paths.
- **`@` binding interpolation is wired against the PARENT scope, not `attrs.$observe`.** AngularJS-canonical: consumer markup like `<my-dir title="{{outerName}}">` must resolve `outerName` in the outer namespace, but `attrs.$$scope` after `bindAttrsToScope` is the ISOLATE scope. Reaching for `attrs.$observe` would install the interpolation watch against the isolate scope, leaving `{{outerName}}` evaluating to undefined. Instead the wiring resolves `interpolate(rawAttr, true)` once, installs a `scope.$watch` ON the isolate scope that evaluates the interpolation against the parent scope, and writes the result to the target. Tear-down still rides on the isolate scope's `$destroy()`. This diverges from the brief's suggestion to "just call `attrs.$observe(attrName, …)`" — that approach would silently break interpolated `@` bindings.
- **`wireIsolateBindings` accepts an optional `interpolate` arg.** When omitted (standalone unit-test callers) the `@` binding degrades to a one-time seed of the raw attribute string; the watcher is suppressed. The compiler's two call sites pass the resolved `$interpolate` service threaded through `CompileOptions`.
- **`=` two-way write-back uses a local `writeAssignable` helper.** The parser's `assign` is module-internal; rather than widening the parser's public surface for one Slice-1 consumer, the helper re-implements the narrow AST-walk needed (`Identifier` / `MemberExpression` only). Non-assignable parent expressions silently degrade to one-way — this matches AngularJS semantically (it would throw `$compile:nonassign` but we choose not to introduce a new error class for that edge case in Slice 1).
- **`=*` collection-mode is REJECTED.** The brief allows accepting it as plain `=`; the regex deliberately rejects it via `InvalidIsolateBindingError`. Collection mode is rare and our tests don't exercise it; deferring it to a future spec keeps the binding contract narrow.
- **Conflict-guard early-return on `MultipleIsolateScopeError`.** When two object-form-scope directives match on the same element, the per-element linker routes the error via `$exceptionHandler('$compile')` and RETURNS before any binding / pre-link / post-link / child-link wiring runs. The directive's other behavior (link / compile / etc.) is intentionally skipped for the conflicting element; sibling elements continue to link normally.
