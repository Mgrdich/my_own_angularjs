## Notes for the Implementation Agent

- **`import type` only.** The new imports from `@compiler` / `@controller` MUST be type-only — a non-type import creates a real `@di → @compiler` / `@di → @controller` runtime cycle. The `.filter` precedent (`import type { FilterFn, IFilterProvider } from '@filter/filter-types'`) is the reference.
- **Runtime method on `Module`, typed overloads on `TypedModule`.** Never put `Registry`-dependent parameter types on the `Module` class — it breaks the `Module<A> <: Module<B>` variance that `AnyModule` / `createInjector` rely on. The `.filter` method is the reference split.
- **No new files, no build/export changes.** The entire feature is `src/di/module.ts` + tests + docs.
- **Parity-by-construction.** The DSL methods own no state and add no validation — accumulation (`.directive`), last-wins (`.controller`), error messages, and timing are all inherited from the providers. Tests verify the inheritance, not re-implement it.
- **`.component` is OUT** — deferred to the "Components & isolate scope" roadmap item (it needs isolate scope, `bindToController`, lifecycle hooks). Do not add a `.component` method.
