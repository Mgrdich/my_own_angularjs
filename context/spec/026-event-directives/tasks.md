## Notes for the Implementation Agent

- **No new `EXCEPTION_HANDLER_CAUSES` entry.** The `'eventListener'` token (already in the 10-tuple from spec 002's scope events) is the natural fit for native-event-listener throws. The `$apply` exception path may route via a different existing token — verify during implementation, but no new entry needed regardless. The tuple stays at 10.
- **`EventName` type is the compile-time safety net.** `EVENT_NAMES as const satisfies readonly (keyof HTMLElementEventMap)[]` rejects typos at type-check time. The 18 `createEventDirective('…')` call sites pass type-checked string literals. A future spec adding a new event directive extends the tuple; the type narrows automatically.
- **`$$phase`-aware dispatch is load-bearing.** Without it, a nested event (one event firing during another's `$apply`) throws "$digest already in progress." The check + `$evalAsync` fallback is AngularJS-canonical.
- **`parse(attrs.ngXxx)` at compile time, NOT at event time.** Parsing 18 expressions per (element × directive) at link time is acceptable; parsing per event fire would be wasteful. The link fn closes over the parsed function.
- **All 18 directives in ONE file.** `ng-event-directives.ts` mirrors AngularJS-1.x's `ngEventDirectives` pattern — one file, one factory helper, 18 outputs.
- **No exports from `@compiler/index`.** DI-registration only — same precedent as `ngTransclude` (spec 018) and all spec 023/024/025 directives.
- **Tick task checkboxes in the SAME commit as the implementation** per `CLAUDE.md` — the co-modification of `tasks.md` + `src/` in one commit is the traceability signal the audit looks for.
