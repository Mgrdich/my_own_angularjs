  - [x] Add `.filter<K extends string, F extends FilterFn>(name: K, factory: Invokable<F>)` to the `Module` class in `src/di/module.ts`. **[Agent: typescript-framework]**
    - **DEVIATION (2026-05-07):** The brief asks for a dual-write — a `<name>Filter` provider shim plus a `$filterProvider.register(...)` config block. Implementing it that way produced a divergence: `injector.get('<name>Filter')` resolved through the shim's `$get`, while `$filter(name)` resolved through `$filterProvider`'s private map — so a decorator on `<name>Filter` was visible only on the provider-shim path, not through `$filter('<name>')`. FS §2.6 requires decorators to be visible on BOTH paths. The implementation pivots to the AngularJS-canonical architecture: `$filterProvider.register(name, factory)` routes through `$provide.factory(name + 'Filter', factory)`, making each filter a normal injector-resolvable factory. `$filter(name)` returns `$injector.get(name + 'Filter')`. `module.filter()` becomes pure sugar — it pushes ONE coordinated record (a config block invoking `$filterProvider.register`); the `<name>Filter` provider entry materialises through the shared `applyRegistrationRecord` machinery. Decorators reach filters with no extra wiring; last-wins works uniformly across `.filter`, `$filterProvider.register`, and `$provide.factory` because all three writers share the same factory map.

With the new architecture, `module.filter()` writes the producer through a config block (via `$provide.factory`), so the validation pass needs to see those writes.
