# my-own-angularjs â€” AI context

A from-scratch TypeScript reimplementation of AngularJS. Goal: **clarity over performance**. Every feature goes through the AWOS spec-driven workflow before code.

## Commands

```bash
pnpm test          # vitest (jsdom env, 90% line-coverage threshold)
pnpm typecheck     # tsc --noEmit, strict + noUncheckedIndexedAccess

pnpm format        # Prettier (semi, singleQuote, printWidth 120)
pnpm build         # Rollup dual ESM+CJS with .d.ts for each subpath
```

CI (`.github/workflows/ci.yml`) gates on: lint â†’ format:check â†’ typecheck â†’ test. `build` is not yet in CI.

## Non-obvious invariants

- **No `new Function()` / no `eval()`** â€” the expression parser uses a **tree-walking interpreter** deliberately. This avoids CSP violations and is part of the project's security posture. Don't "optimize" by generating code strings.
- **Digest TTL contract** â€” configurable via `Scope.create({ ttl: 20 })` (default 10). TTL breach throws with the watch function source in the error to help identify unstable watchers.
- **One-time bindings & constant watches** (spec 010) â€” the parser attaches `literal` / `constant` / `oneTime` flags on AST nodes. Scope wires `oneTimeWatchDelegate` (literal) or `oneTimeLiteralWatchDelegate` for `::expr` expressions; constant expressions use `constantWatchDelegate`. When modifying the watcher wiring, preserve those delegate selections.
- **Module boundary rule** â€” `parser/*` and `di/*` depend only on `@core` (prefer importing from `@core/index`, not `@core/utils` directly). `core/scope.ts` intentionally depends on `@parser/index` because scopes evaluate expression strings. `compiler/` has no dependencies yet.
- **Error handling in digest** â€” listener/watch exceptions are logged with `console.error('...', e)` and the digest continues. Don't swallow silently and don't abort the digest loop on a single listener failure.
- **Strict mode is frozen after the config phase** â€” `$sceProvider.enabled(false)` is the only way to disable SCE. Once the injector finishes config, `$sce.isEnabled()` is permanent. Strict-OFF turns both `trustAs*` and `getTrusted*` into total pass-throughs (no wrapper classes are constructed).
- **Trusted values are per-context nominal classes** â€” `TrustedResourceUrl extends TrustedUrl`, so a trusted resource URL is accepted where a trusted URL is expected (not vice-versa). Identity is checked via `instanceof`, not a string-based brand. Do NOT "optimize" to a single branded wrapper â€” the subtype rule matters for AngularJS parity.
- **`ngSanitize` is opt-in, never registered on the core `ng` module** â€” apps must list `'ngSanitize'` in their dependency chain. When loaded, `$sce.getTrustedHtml(plainString)` automatically routes through `$sanitize` via a lazy `$injector.has('$sanitize')` lookup in `$SceProvider.$get` â€” no hard dependency from `$sce` to `ngSanitize`, no decoration. Removing the `$injector` dep from `$SceProvider.$get` would silently break this integration; the regression test in `src/sanitize/__tests__/sanitize-sce.test.ts` is the guard.
- **The digest's "log and continue" contract is preserved through `$exceptionHandler`.** A failing watcher / listener / async task is reported via the configured handler and the digest proceeds; only TTL exhaustion re-throws (after first reporting via the handler, cause `'$digest'`). The default handler is `console.error`, so today's logs continue to appear unchanged. A custom handler that itself throws is caught by `invokeExceptionHandler` and degrades to `console.error` â€” the digest still does not crash. The nine `EXCEPTION_HANDLER_CAUSES` tokens are part of the public contract; future specs that add framework-internal call sites must extend the list as a public-API change.
- **`$provide` is config-phase only** â€” the six methods (`factory`, `service`, `value`, `constant`, `provider`, `decorator`) throw synchronously with `$provide.<method> is only callable during the config phase; calling it after the run phase begins is not supported` whenever they're invoked outside a `config()` block, including via a `$provide` reference captured during config and called later. This out-of-phase error is a programming error and is **not** routed through `$exceptionHandler` â€” it surfaces directly to the caller. Constants are protected by an override guard: `value` / `factory` / `service` / `provider` / `decorator` (whether through the module DSL or `$provide`) targeting a name already registered as a `.constant` throw `Cannot override constant "<name>" â€” already registered via .constant(...)`. Within the unified registration timeline, a new producer recipe wipes prior producer entries for the same name from the other backing maps so the run-phase resolver returns the most-recent producer's value; decorators stack on the current producer and are NOT evicted.
- **Filters internally registered as `<name>Filter` providers.** `module.filter('currency', factory)` is functionally equivalent to `module.provider('currencyFilter', { $get: factory })` â€” `injector.get('currencyFilter') === $filter('currency')` (same reference). The `<name>Filter` naming makes decorators work for free: `module.decorator('currencyFilter', ['$delegate', â€¦])` wraps the underlying filter and is visible through both lookup paths. The `$stateful = true` flag on a filter function (not the factory) opts the filter out of the digest's input-identity short-circuit; all nine built-ins are stateless. `$locale` is a single `module.factory('$locale', () => myLocale)` swap â€” `currency`/`number`/`date` read it lazily on each invocation, so config-time replacement takes immediate effect at run-time. Unknown filters surface as `Unknown filter: <name>` (a `FilterLookupError` instance) routed through `$exceptionHandler` with cause `'$filter'` at digest time; the digest continues. `parse(expr)` conservatively marks any filter-containing expression as `constant: false` at parse time; scope re-checks at watch install once `$filter` is reachable, upgrading to a constant-watch when the input/args are structurally constant AND no filter in the chain is `$stateful`.
- **Directive registration accumulates per name (no last-wins).** Two `$compileProvider.directive('foo', factoryA)` + `.directive('foo', factoryB)` calls produce TWO directives that both run on `<div foo>`. Mirrors AngularJS exactly. For services / filters / providers the rule is last-wins; for directives it is accumulation. Internally each name is a `<name>Directive` provider whose `$get` returns the array of normalized directive objects, so `injector.get('fooDirective')` returns both factories' resolved entries. Decorators on `<name>Directive` wrap the WHOLE array â€” the `$delegate` is the array, and the decorator may filter or wrap individual entries.
- **`.directive` / `.controller` / `.filter` module-DSL methods are pure config-block sugar (spec 021 / 016).** Each pushes ONE config block onto the module's `$$configBlocks` that forwards verbatim â€” `.directive` â†’ `$compileProvider.directive`, `.controller` â†’ `$controllerProvider.register`, `.filter` â†’ `$filterProvider.register`. The DSL owns no registry and adds no validation: accumulation (`.directive`), last-wins (`.controller`), error messages, and registration timing are all inherited from the providers â€” tests verify the inheritance, they don't re-implement it. The `@di â†’ @compiler` / `@di â†’ @controller` references in `src/di/module.ts` are **`import type`-only** (erased at build), so `@di` keeps zero runtime dependency on `@compiler` / `@controller` and the module-boundary invariant holds â€” `.filter`'s `import type { â€¦ } from '@filter/...'` is the established precedent. **Registry-widening asymmetry:** single-name `.directive('x', â€¦)` widens the typed `Registry` with `xDirective: Directive[]` (the `<name>Directive` provider is genuinely injector-resolvable, like `<name>Filter`), but bulk-map `.directive({â€¦})` and BOTH `.controller` forms do NOT widen â€” controllers live in `$ControllerProvider`'s private `$$registry` Map and are never injector-resolvable, so there is no key to widen with. As with `.filter`, the runtime methods live on the `Module` class while the dependency-typed overloads live on the `TypedModule` interface â€” the same class-vs-interface split that avoids putting `Registry` in a contravariant position and breaking `Module` subtype variance.
- **Compile-phase mutation runs once per template.** Compile fns run during the walker's recursive descent and return link functions; the same compiled subtree can be linked against different scopes by re-invoking the linker, but compile runs exactly once. Directives that need template-time DOM mutation (adding classes, setting attributes that affect child compilation) put it in `compile`, not `link`. The `Attributes` instance is created during compile and shared across every link invocation â€” state stored on `attrs` (e.g. observers from `$observe`) accumulates across linker calls, so re-linking the same template against multiple scopes is supported but uncommon.
- **`$observe` lazily installs the per-attribute watch on first observer registration.** A directive that calls `attrs.$observe('href', fn)` triggers `$interpolate(attrs.href, true)` exactly once; if interpolated, a single `scope.$watch(interpolateFn, listener)` is installed and reused for all subsequent observers of the same attribute name. Static attributes get a one-shot `$evalAsync` notification per observer â€” no watch is wired. Without `$observe` registrations, no per-attribute watches exist at all, so an element with hundreds of static-looking attributes costs zero digest time.
- **`'$compile'` cause token is the 10th `EXCEPTION_HANDLER_CAUSES` entry.** Public-API additive change introduced in spec 017 (the `'$filter'` precedent from spec 016 is the prior pattern). Used for directive factory invocation (lazy at `<name>Directive` provider `$get`), compile, pre-link, post-link, and the `$set`-driven `$observe` callback notification path. The static-attribute `$observe` first-fire (scheduled via `$evalAsync`) routes through the standard `'$evalAsync'` cause â€” both surfaces are documented and tested.
- **Isolate scope intentionally created via `Scope.$new(true)` (the existing core API).** The compiler does NOT reimplement isolate scope â€” `parentScope.$new(true)` fires once per element bearing a `scope: { â€¦ }` directive; two such directives on the same element throw `MultipleIsolateScopeError` routed via `$exceptionHandler('$compile')` from the per-element linker. A `bindToController: {â€¦}`-only directive does NOT create its own isolate scope (it consumes whatever scope the element already has), so it cannot trigger the multi-isolate conflict. `scope: false | true` from spec 017 still works unchanged; the per-element linker discriminates "child" vs "isolate" by checking `isolateBindings != null` on the normalized directive.
- **Raw `Element` argument is the deliberate choice.** No jqLite shim ships in spec 017; the link signature is `(scope, element, attrs)` where `element` is a native `Element` â€” or a `Comment` for an M-restricted match. The Phase 5 `angular.element` compat layer may layer on top without changing the link signature, so directives written today against raw `Element` will keep working when jqLite-style sugar lands.
- **Child-scope cleanup via private element properties.** When `scope: true` is requested, the compiler stashes the child scope on `(element as any).$$ngScope` and lets directives push cleanup callbacks onto `(element as any).$$ngCleanupQueue` â€” both non-enumerable. Future structural directives (`ng-if`, `ng-repeat`, â€¦) MUST call `destroyElementScope(element)` before removing nodes; there is no `MutationObserver` or auto-detection. Without that call, removed-subtree scopes stay attached to the parent's watcher tree forever.
- **Transclusion scope is a child of the OUTER scope, not the directive's.** The scope the directive was linked against (`parentScope` inside the per-element linker) is captured by the `$transclude` closure BEFORE the `scope: true` `$new()` runs, so every clone's transclusion scope is `parentScope.$new()` â€” never `directiveScope.$new()`. This is what lets `<my-card scope="true">` keep its own variable namespace internally while consumer markup inside `<my-card>` still binds to OUTER variables.
- **Captured children compile EXACTLY ONCE at capture time.** Capture runs during the host directive's compile phase, before the walker descends into the host's child positions, so the OUTER walker sees no children to link. Each `$transclude(...)` call deep-clones the master via `Node.cloneNode(true)` and re-links the clone against a fresh transclusion scope through the internal `cloneMap` indirection (the public `Linker` type is unchanged). The master fragment is never inserted into the DOM.
- **Multi-clone is supported.** Each `$transclude(...)` call yields an independent clone with an independent scope, pushed onto the host element's `$$ngCleanupQueue` so `destroyElementScope(host)` tears every clone down regardless of how many were produced. This is the infrastructure `ng-repeat` will lean on. The outer scope's `$destroy()` also tears clones down via normal scope-tree propagation â€” both paths converge on `transcludedScope.$$destroyed === true`.
- **`ng-transclude` finds its enclosing transcluding directive by walking `element.parentElement`** until it finds an ancestor carrying the non-enumerable `$$ngBoundTransclude` slot. Decoupled from the walker â€” works for programmatically-inserted markers AND (in the future) for markers inside compiled `template` strings. The slot stashes `{ fn, declaredSlots, kind, directiveName }` so the marker validates slot-name lookups before invoking `$transclude`.
- **`transclude: 'element'` is deliberately deferred and REJECTED at registration** so a future structural-directives spec can add it without a silent semantic change. The `'$compile'` cause is reused for every transclusion error site â€” no new `EXCEPTION_HANDLER_CAUSES` entry (the tuple stays at 10). The nine new error classes are exported from `@compiler/index`; registration-time errors route via the existing factory try/catch in `$$buildDirectiveArrayProvider`, link-time errors route via direct `invokeExceptionHandler(handler, err, '$compile')` calls.
- **Template install slots AFTER transclude capture + BEFORE the per-directive compile loop on the host element.** The `compileElementOrComment` pre-pass detects the template-declaring directive AFTER spec 018's transclude capture has stashed `$$ngBoundTransclude` on the host, then replaces the host's children with the resolved template nodes BEFORE the matched-directive `compile` functions run. As a result the canonical `transclude: true` + `template` + `<ng-transclude>` wrapper pattern works without extra wiring â€” the consumer's children are already captured when the template overwrites them, and `<ng-transclude>` markers inside the template find the host's `$$ngBoundTransclude` via the existing `parentElement` walk.
- **`$compile`'s public `Linker` signature is unchanged in spec 019.** `(scope: Scope) => Element | NodeList | Comment` is preserved even when `templateUrl` directives are present in the compiled subtree. Async resolution happens internally â€” the top-level `$compile` schedules `Promise.resolve().then(drainDeferredTemplateQueue)` after the synchronous walker returns. Spec-017/018 callers remain assignable. Tests that observe post-install DOM `await Promise.resolve()` at least twice (defensively three times) to flush the drain chain.
- **`$templateRequest` deduplicates concurrent fetches via an `inFlight: Map<string, Promise<string>>`.** Two requests for the same URL before the first resolves share a single network call (and the cache-write side effect on resolution). Cleanup happens after settlement so a subsequent request after `$templateCache.remove(url)` re-fetches. Tests that exercise the deduplication seam usually inject a deferred mock fetcher via `createTemplateRequest({ cache, fetcher })`.
- **`replace: true` is REJECTED permanently** at registration via `ReplaceTrueNotSupportedError`. AngularJS 1.x officially deprecated it; this project will NOT ship it. Templates always become the host element's children â€” the host element itself (tag, attributes, listeners) is preserved. `replace: false` and `replace: undefined` are accepted; any other value also throws `ReplaceTrueNotSupportedError`.
- **Function-form templates are called EXACTLY ONCE per compile invocation per host element.** The resolved string is memoized on the directive's local entry; subsequent `linker(scope1); linker(scope2)` invocations against the same compiled tree reuse the computed string. This applies equally to `template: (el, attrs) => string` and `templateUrl: (el, attrs) => string`. A function-form template that throws or returns a non-string value routes via `$exceptionHandler('$compile')` (`TemplateFunctionReturnedNonStringError` / `TemplateUrlFunctionReturnedNonStringError`) and leaves the host element empty.
- **`'$compile'` cause token is reused for every template-related error site.** The `EXCEPTION_HANDLER_CAUSES` tuple stays at 10 entries; spec 019 introduces NO new cause token. Template-fetch failures, multi-template-on-element, function-form throws / non-string returns, registration-phase validation errors (`InvalidTemplateValueError`, `InvalidTemplateUrlValueError`, `EmptyTemplateError`, `EmptyTemplateUrlError`, `TemplateAndTemplateUrlCombinedError`, `ReplaceTrueNotSupportedError`), and link-phase multi-template detection (`MultipleTemplateDirectivesError`) all route through the same handler the rest of `$compile` uses. A custom handler that itself throws is caught by spec-014's `invokeExceptionHandler` recursion guard and falls back to `console.error` â€” template loading does NOT crash.
- **Controllers register through `$controllerProvider.register(name, fn)`, instantiate through `$controller(nameOrFn, locals?, ident?)`.** The single alias parser `CONTROLLER_NAME_ALIAS_RE = /^(\S+?)(\s+as\s+([\w$]+))?\s*$/` and the shared `IDENT_RE = /^[A-Za-z_$][\w$]*$/` are exported from `@controller/controller.ts` so the directive-side `controllerAs` validation in `normalizeDirective` uses the same source of truth as `parseControllerName`; an explicit `ident` argument supersedes the `'Foo as vm'` alias suffix when both are present. `$controllerProvider.register` is config-phase only â€” the captured `$$getPhase` thunk on `ProvideService` (`$provide.$$getPhase`, an internal hook surfaced in spec 020 for cross-provider use â€” app code MUST NOT call it) trips the guard even on a `$controllerProvider` reference saved during config and called from a `run()` block; `$controller` itself is run-phase only, reachable via `injector.get('$controller')` after `createInjector` finishes. The compiler's per-element controller seam runs ONCE per directive declaring `controller`, AFTER `$transclude` setup and BEFORE the directive's own pre-link (the shared `runControllerSeam` helper closes over `$controller` + `exceptionHandler` and fires from BOTH the inline-link path and the post-template-install path inside `buildPostTemplateLinker`, so `templateUrl` directives still see their controller at the same point in the post-template link cycle); throws route via the existing `'$compile'` cause â€” no new `EXCEPTION_HANDLER_CAUSES` entry, the tuple stays at 10. `controllerAs`-without-`controller` is REJECTED AT DIRECTIVE REGISTRATION (in `normalizeDirective`'s `normalizeController` block, routed via the existing factory `try/catch` in `$$buildDirectiveArrayProvider` through `$exceptionHandler('$compile')`), NOT at link time. Instantiation follows AngularJS-canonical `Object.create(prototype)` + `injector.invoke` + return-value replacement â€” a constructor returning a non-null object REPLACES the prototype-instance (matches `$injector.instantiate`). `register(name, fn)` is last-wins on duplicates (mirrors services / filters; contrasts with directives' accumulation). The direct-call path does NOT route exceptions through `$exceptionHandler` â€” `$controller('Bad', {})` propagates `UnknownControllerError` to the caller; the compile-time path routes the SAME error via `$exceptionHandler('$compile')`. This asymmetry mirrors AngularJS 1.x exactly: direct callers own their `try/catch`. Bare-function controllers without `$inject` will throw at instantiation because `injector.invoke` rejects un-annotated functions â€” there is no source-parsing fallback. The array-wrap form `['$scope', function ($scope) { ... }]` is the canonical minification-safe shape used everywhere in the suite. ES classes throw `TypeError` when invoked without `new`; wrap them in a factory function (`['$scope', ($s) => new MyClass($s)]`). The `'hasOwnProperty'` name is reserved at both registration and lookup as a prototype-pollution guard.
- **The four binding-wiring strategies are NOT symmetric.** `=` installs both a parent â†’ local watch AND a reverse local â†’ parent watch (the reverse path is suppressed for non-assignable parent expressions â€” non-assignable `=` silently degrades to one-way, deliberately NOT throwing `$compile:nonassign`); the last-digest-value reconciliation guards against ping-pong loops. `@` does NOT use `attrs.$observe` â€” it interpolates against the PARENT scope (not the isolate) via `interpolate(rawAttr, true)` and installs the watcher ON the isolate scope so `<my-dir title="{{outerName}}">` resolves `outerName` in the outer namespace; this is a deliberate divergence from a naive `attrs.$observe` reading, which would resolve against the isolate scope (where `outerName` does not exist). `<` synchronously seeds the local at link time before the watcher runs (so `$onInit` sees a populated value), then installs a single one-way watch with the first-fire `newValue === oldValue` sentinel suppressed in the `onChange` callback. `&` is a closure over `parentScope.$eval(expr, locals)` â€” no watcher; the target's slot value IS the callable.
- **`later: true` instantiation ordering is fixed.** `createController(name, locals, ident, true)` returns `{ instance, identifier }` and defers `bindAlias`. The compiler's per-element seam then runs: construct â†’ stash in `$$ngControllers` â†’ resolve `require` (object-form auto-assigns onto the requiring controller's instance) â†’ wire bindings (target = controller instance when `bindToController` is in effect) â†’ publish the `controllerAs` alias on the isolate scope â†’ `$onInit` â†’ preLink â†’ child link â†’ postLink â†’ `$postLink` â†’ `$onDestroy`. A shared-spy ordering test pins this. The 1â€“3 arg `$controller(...)` call sites still publish the alias internally (additive, no breaking change). `later: true` is the ONLY valid value for the 4th arg â€” `later: false` is intentionally NOT overloaded as a 4-arg shape (it would be indistinguishable from omitting the arg).
- **`$onChanges` is wired ONLY through the `bindToController` instance-target path.** The scope-target wiring deliberately does NOT pass an `onChange` callback to `wireIsolateBindings`. `$onChanges` fires SYNCHRONOUSLY once at link time with all `<` and `@` bindings marked `isFirstChange()` = true; subsequent changes batch through a per-digest `$$postDigestQueue` flush via `ChangesQueue.record(...)` + `flushChangesQueue(...)`. `=` and `&` bindings NEVER feed `$onChanges` â€” one-way data flow only. `previousValue` on the initial fire is the frozen `UNINITIALIZED_VALUE` sentinel (`Object.freeze(Object.create(null))`); consumers should gate on `isFirstChange()` instead of reading it. A controller's `$onDestroy` calls `queue.clearForController(ctrl)` so a pending flush after destroy does NOT fire.
- **`$$ngControllers` element stash** â€” each element bearing a controller gets a non-enumerable `$$ngControllers: Map<string, unknown>` (spec 022 Slice 3), parallel to spec-018's `$$ngBoundTransclude` and spec-017's `$$ngScope`. `require` resolution (Slice 4) reads exclusively from this map; no `MutationObserver` or DOM query is involved. Future structural directives MUST keep populating this slot when they relocate controlled elements, or `require: '^foo'` lookups will silently miss.
- **`require` form-aware semantics.** `resolveRequireForm(element, requiringName, requireSpec)` dispatches on the runtime shape: string / array / object. Flag parsing is order-tolerant â€” `'?^name'` and `'^?name'` both yield `{ prefix: '^', optional: true, name: 'name' }`; `^^` is consumed before `^` so longer prefixes win. `^` walks own element then `parentElement`; `^^` walks `parentElement` only; `?` makes a miss yield `null` instead of throwing `MissingRequiredControllerError`. Object form is the ONLY shape that auto-assigns onto the requiring controller's instance BEFORE `$onInit`; string and array forms deliver controllers exclusively via the link fn's 4th argument. The `LinkFn.controllers` slot was reserved as `undefined` since spec 018 and is widened to `unknown` in Slice 4 â€” existing 3-arg link signatures remain assignable. The directive's other behavior (link / compile / transclude) on the same element still runs on a non-optional miss; sibling elements are unaffected.
- **`$compileProvider.component(name, definition)` translation.** The CDO becomes a directive factory `[() => DDO]` (array-wrapped to satisfy strict `annotate` â€” bare functions without `$inject` are rejected) with defaults `restrict: 'E'`, `scope: definition.bindings ?? {}` (object-form â†’ isolate scope, possibly empty), `bindToController: true`, `controller: definition.controller ?? [function NoopController() {}]`, `controllerAs: definition.controllerAs ?? '$ctrl'`, pass-through `template` / `templateUrl` / `transclude` / `require`. The empty `NoopController` keeps stack traces clean; the array wrap is the canonical zero-dep shape used everywhere else in the suite. `.component` is single-name only (no bulk-map form â€” AngularJS 1.x never had one) and is non-widening on `TypedModule` (matches `.controller`; contrasts with single-name `.directive`'s `${K}Directive: Directive[]` widening â€” the directive provider IS injector-resolvable, the component is not).
- **`EXCEPTION_HANDLER_CAUSES` STILL at 10 after spec 022.** Spec 022 introduces FOUR new error classes (`InvalidIsolateBindingError`, `MultipleIsolateScopeError`, `MissingRequiredControllerError`, `InvalidComponentDefinitionError`) AND four new lifecycle-hook throw sites (`$onInit` / `$onChanges` / `$onDestroy` / `$postLink`) AND the `require` resolution failure site â€” every one of them reuses the existing `'$compile'` cause token. No new entry to the `EXCEPTION_HANDLER_CAUSES` tuple. The two `.component`-level name + plain-object validations (`InvalidComponentDefinitionError`) surface SYNCHRONOUSLY to the caller (matching `.directive`'s name / factory validation precedent), NOT via `$exceptionHandler`. Downstream errors at provider `$get` time (`InvalidIsolateBindingError` from `parseIsolateBindings`, `InvalidControllerFactoryError`, â€¦) still route lazily via the existing factory `try/catch` in `$$buildDirectiveArrayProvider`.
- **`IsolateScopeNotSupportedError` is RETIRED but exported as `@deprecated`.** Spec 022 Slice 1 removed the throw site in `normalizeDirective`; the class itself stays exported from `@compiler/index` and the root barrel for a one-release deprecation grace period so a consumer catching it via `instanceof IsolateScopeNotSupportedError` keeps compiling without a sudden `ReferenceError`. The two re-export sites carry inline `eslint-disable @typescript-eslint/no-deprecated` justifications. A future spec may remove the class outright. Use `InvalidIsolateBindingError` for malformed binding specs and `MultipleIsolateScopeError` for the two-isolate-directives conflict.
- **`terminal: true` halts CHILD-node recursion ONLY for `ng-non-bindable`** (spec 023 Slice 1, narrowed). Spec 017's broader same-element terminal cutoff applies to every `terminal: true` directive, but the no-descent extension is gated on `directive.name === 'ngNonBindable'`. The narrowing avoids breaking the spec 017 test contract (`src/compiler/__tests__/terminal.test.ts:178` pins the original semantic for custom `terminal: true` directives, i.e. "terminal does NOT affect descendants"). Future structural directives (`ng-if`, `ng-repeat`) will get no-descent via dedicated mechanisms (`transclude: 'element'`), not via this hook. The hook lives in `src/compiler/compile.ts` in `compileElementOrComment`, gated on `effectiveDirectives.some(d => d.terminal && d.name === 'ngNonBindable')`.
- **The seven spec-023 built-in directives are registered on `ngModule` but NOT exported from `@compiler/index`.** `ngCloak`, `ngBind`, `ngBindTemplate`, `ngBindHtml`, `ngShow`, `ngHide`, `ngNonBindable` â€” all reachable via `injector.get('<name>Directive')` when an app's module declares `'ng'` in its deps chain, but the factory functions themselves are file-local exports (not in the root barrel or `@compiler/index`). Matches the spec-018 `ngTransclude` precedent. Consumers swap them via `module.decorator('<name>Directive', â€¦)` or override-registration via `module.directive('<name>', â€¦)`. `EXCEPTION_HANDLER_CAUSES` STILL at 10 after spec 023 â€” every error site reuses the existing `'$compile'` or `'watchListener'` tokens. The `ng-bind-html` directive consumes the existing spec-013 `$sce` â†’ `$sanitize` integration and does not re-implement sanitization â€” apps swap sanitizers via `module.decorator('$sanitize', â€¦)`, transparently visible through the `ng-bind-html` listener. The `ng-show` / `ng-hide` / `ng-cloak` directives depend on consumer-shipped CSS (`.ng-hide { display: none !important; }` + `[ng-cloak], .ng-cloak { display: none !important; }`); the framework never auto-injects a stylesheet. `$animate` integration is a Phase 4 roadmap item â€” visibility transitions are synchronous today.
- **`ng-style` dispatches on hyphen-presence: kebab-case keys go through CSSOM's `setProperty`/`removeProperty`, camelCase keys go through direct IDL assignment.** CSSOM's `setProperty` is specified to accept ONLY kebab-case property names per W3C; jsdom enforces this strictly. So `<div ng-style="{ 'background-color': 'red' }">` (kebab) routes through `setProperty('background-color', 'red')` while `<div ng-style="{ backgroundColor: 'red' }">` (camel) routes through `element.style.backgroundColor = 'red'`. Both forms work in tests and production; neither uses `cssText`, so the consumer-shipped-style preservation guarantee holds for both paths. Matches AngularJS 1.x's own `$$style.js` implementation. The hook lives in `src/compiler/ng-style.ts` â€” search for the `applyStyle` / `clearStyle` helpers. `EXCEPTION_HANDLER_CAUSES` STILL at 10 after spec 024 â€” every directive's diff cycle runs inside a `$watchCollection` listener and reuses the existing `'watchListener'` token.
- **`installClassWatcher(scope, element, expr, gate?, gateProperty?)` is the shared engine** powering `ng-class`, `ng-class-even`, and `ng-class-odd` (all in `src/compiler/ng-class.ts`). When `gate` and `gateProperty` are both provided, the engine ALSO installs a secondary `scope.$watch(gateProperty, refire)` so a `$even` / `$odd` flip re-runs the diff against the cached `lastValue` (the previously-seen expression value). The closure tracks `appliedClasses: Set<string>` for the classes-preserved guarantee â€” only classes WE added are eligible for removal. Consumer-shipped classes (e.g. `<div class="card" ng-class="â€¦">`) never enter `appliedClasses` and never get removed. Mirrors `ng-style`'s `appliedProps: Set<string>` mechanism but on the `classList` surface. The shared module-private `flattenClassExpression` helper (`src/compiler/class-expression.ts`) normalizes the three expression forms (string / array / object) into a single class-name set and is consumed only by `ng-class.ts` â€” not exported from `@compiler/index`.
- **`attrs.$set(name, value)` removes the attribute ONLY when `value === null`, NOT for any falsy value.** The signature is `(name: string, value: string | null, writeAttr?: boolean)`. An empty string `''` is treated as a valid attribute value (`setAttribute(name, '')` produces bare-presence form like `<button disabled>`); only `null` triggers `removeAttribute`. Spec 025's URL helper maps empty/undefined to `null` explicitly via `value !== undefined && value !== '' ? value : null`; the boolean helper maps `value ? '' : null` (empty string is the canonical presence form, equivalent to bare-presence per HTML5). Any future directive that writes attributes through `$set` must observe this contract â€” passing `false` does NOT remove the attribute, it calls `setAttribute(name, false)` which the browser serializes as `name="false"` (cosmetic noise without removal), and is a type error besides because the public signature rejects boolean. The hook lives in `src/compiler/attributes.ts` â€” search for the `$set` body around `attributes.ts:271` (the `value === null` branch).
- **The eight spec-025 attribute helper directives are registered on `ngModule` but NOT exported from `@compiler/index`.** `ngHref`, `ngSrc`, `ngSrcset` (URL aliases, priority 99) + `ngDisabled`, `ngChecked`, `ngReadonly`, `ngSelected`, `ngOpen` (boolean aliases, priority 100) â€” all reachable via `injector.get('<name>Directive')` when an app declares `'ng'` in its deps chain. Matches the spec-018 / spec-023 / spec-024 DI-only precedent (factory functions are file-local exports in `src/compiler/ng-attribute-aliases.ts`; both module-private factory helpers `createUrlAliasDirective` and `createBooleanAliasDirective` are NOT exported either). The URL aliases rely on the framework's compile-then-digest ordering for the "pre-compile attribute absent" guarantee (the real `href` / `src` / `srcset` attribute is never present before the first digest fires, so a browser click before compile goes nowhere instead of to the literal `"{{url}}"`). The boolean aliases produce bare-presence DOM via the `$set` empty-string mapping. No new error classes, no new `EXCEPTION_HANDLER_CAUSES` token â€” the tuple stays at 10. Errors thrown from the `$observe` listener (URL pattern) or `$watch` listener (boolean pattern) flow through the existing `'watchListener'` / `'$evalAsync'` causes.
- **`scope.$apply` lacks the upstream AngularJS try/catch.** This project's `$apply` (at `src/core/scope.ts:486-494`) wraps the runner in `try/finally`, NOT `try/catch` â€” a throw inside the runner propagates past `$apply` and is never reported via `$exceptionHandler`. Upstream AngularJS catches the throw inside `$apply` and routes it via `$exceptionHandler(e)`. Spec 026's event directives compensate by wrapping their own `$apply`/`$evalAsync` dispatch in a `try/catch` and routing via `invokeExceptionHandler($exceptionHandler, err, 'eventListener')`. Any future directive that calls `scope.$apply` directly must observe the same workaround OR a future "scope $apply parity" spec patches `$apply` to mirror upstream (which would make the directive's `try/catch` redundant but still safe). The hook lives in `src/compiler/ng-event-directives.ts` â€” search for the `try`/`catch` block inside the event handler. Factory signature was widened to `['$exceptionHandler', factory]` (same array-form DI shape that spec 018's `ngTransclude` uses) so the handler is reachable from inside the per-event handler closure.
- **The eighteen spec-026 event directives all register on `ngModule` but NOT on `@compiler/index`.** Mouse (`ngClick`/`ngDblclick`/`ngMousedown`/`ngMouseup`/`ngMouseover`/`ngMouseout`/`ngMousemove`/`ngMouseenter`/`ngMouseleave`), Keyboard (`ngKeydown`/`ngKeyup`/`ngKeypress`), Clipboard (`ngCopy`/`ngCut`/`ngPaste`), Focus (`ngFocus`/`ngBlur`), Form-lifecycle (`ngSubmit`) â€” all reachable via `injector.get('<name>Directive')` when an app declares `'ng'` in its deps chain. Matches the spec 018/023/024/025 DI-only precedent. The shared `createEventDirective(eventName)` helper takes a typed `EventName` (string-literal union derived from `EVENT_NAMES as const satisfies readonly (keyof HTMLElementEventMap)[]` â€” compile-time typo prevention). Each handler parses its expression ONCE at compile time, registers the native listener at link time, and dispatches via `scope.$apply(run)` OR `scope.$evalAsync(run)` based on `scope.$$phase` â€” the `$$phase` check prevents the "$digest already in progress" throw when an event fires during another `$apply`. The `$event` local exposes the native event to the bound expression; locals-first lookup (per spec 009's parser) means it shadows any scope property with the same name. `ng-submit` does NOT auto-`preventDefault` â€” consumers either omit the form's `action` or call it explicitly in the handler. `EXCEPTION_HANDLER_CAUSES.length` stays at 10 â€” every error site reuses the existing `'eventListener'` token (handler throws) or `'$compile'` token (parse-time syntax errors).

## Coding conventions

- **TypeScript strict** (`strict: true` + `noUncheckedIndexedAccess`). No `any` â€” the single existing cast in `src/core/utils.ts:229` is the ceiling, not a precedent.
- **Every `eslint-disable` comment must carry an inline justification** (`-- reason`). CI enforces lint.
- **No explicit return types** when TS inference handles them â€” let inference do the work. Annotate only on exported public-API boundaries where the declared shape is part of the contract.
- **Imports**: use path aliases (`@core`, `@parser`, `@di`, `@compiler`). `no-restricted-imports` blocks `../*` relative climbing.
- **File naming**: kebab-case (`scope-watch-delegates.ts`, `ast-flags.ts`). Tests under `src/<module>/__tests__/*.test.ts`.
- **File size target**: under 500 lines per source file. Refactor candidates today: `src/core/scope.ts` (827), `src/di/module.ts` (776), `src/di/injector.ts` (734).
