# my-own-angularjs — AI context

A from-scratch TypeScript reimplementation of AngularJS. Goal: **clarity over performance**. Every feature goes through the AWOS spec-driven workflow before code.

## Commands

```bash
pnpm test          # vitest (jsdom env, 90% line-coverage threshold)
pnpm typecheck     # tsc --noEmit, strict + noUncheckedIndexedAccess
pnpm lint          # ESLint strictTypeChecked preset
pnpm format        # Prettier (semi, singleQuote, printWidth 120)
pnpm build         # Rollup dual ESM+CJS with .d.ts for each subpath
```

CI (`.github/workflows/ci.yml`) gates on: lint → format:check → typecheck → test. `build` is not yet in CI.

## Non-obvious invariants

- **No `new Function()` / no `eval()`** — the expression parser uses a **tree-walking interpreter** deliberately. This avoids CSP violations and is part of the project's security posture. Don't "optimize" by generating code strings.
- **Digest TTL contract** — configurable via `Scope.create({ ttl: 20 })` (default 10). TTL breach throws with the watch function source in the error to help identify unstable watchers.
- **One-time bindings & constant watches** (spec 010) — the parser attaches `literal` / `constant` / `oneTime` flags on AST nodes. Scope wires `oneTimeWatchDelegate` (literal) or `oneTimeLiteralWatchDelegate` for `::expr` expressions; constant expressions use `constantWatchDelegate`. When modifying the watcher wiring, preserve those delegate selections.
- **Module boundary rule** — `parser/*` and `di/*` depend only on `@core` (prefer importing from `@core/index`, not `@core/utils` directly). `core/scope.ts` intentionally depends on `@parser/index` because scopes evaluate expression strings. `compiler/` has no dependencies yet.
- **Error handling in digest** — listener/watch exceptions are logged with `console.error('...', e)` and the digest continues. Don't swallow silently and don't abort the digest loop on a single listener failure.
- **Strict mode is frozen after the config phase** — `$sceProvider.enabled(false)` is the only way to disable SCE. Once the injector finishes config, `$sce.isEnabled()` is permanent. Strict-OFF turns both `trustAs*` and `getTrusted*` into total pass-throughs (no wrapper classes are constructed).
- **Trusted values are per-context nominal classes** — `TrustedResourceUrl extends TrustedUrl`, so a trusted resource URL is accepted where a trusted URL is expected (not vice-versa). Identity is checked via `instanceof`, not a string-based brand. Do NOT "optimize" to a single branded wrapper — the subtype rule matters for AngularJS parity.
- **`ngSanitize` is opt-in, never registered on the core `ng` module** — apps must list `'ngSanitize'` in their dependency chain. When loaded, `$sce.getTrustedHtml(plainString)` automatically routes through `$sanitize` via a lazy `$injector.has('$sanitize')` lookup in `$SceProvider.$get` — no hard dependency from `$sce` to `ngSanitize`, no decoration. Removing the `$injector` dep from `$SceProvider.$get` would silently break this integration; the regression test in `src/sanitize/__tests__/sanitize-sce.test.ts` is the guard.
- **The digest's "log and continue" contract is preserved through `$exceptionHandler`.** A failing watcher / listener / async task is reported via the configured handler and the digest proceeds; only TTL exhaustion re-throws (after first reporting via the handler, cause `'$digest'`). The default handler is `console.error`, so today's logs continue to appear unchanged. A custom handler that itself throws is caught by `invokeExceptionHandler` and degrades to `console.error` — the digest still does not crash. The nine `EXCEPTION_HANDLER_CAUSES` tokens are part of the public contract; future specs that add framework-internal call sites must extend the list as a public-API change.
- **`$provide` is config-phase only** — the six methods (`factory`, `service`, `value`, `constant`, `provider`, `decorator`) throw synchronously with `$provide.<method> is only callable during the config phase; calling it after the run phase begins is not supported` whenever they're invoked outside a `config()` block, including via a `$provide` reference captured during config and called later. This out-of-phase error is a programming error and is **not** routed through `$exceptionHandler` — it surfaces directly to the caller. Constants are protected by an override guard: `value` / `factory` / `service` / `provider` / `decorator` (whether through the module DSL or `$provide`) targeting a name already registered as a `.constant` throw `Cannot override constant "<name>" — already registered via .constant(...)`. Within the unified registration timeline, a new producer recipe wipes prior producer entries for the same name from the other backing maps so the run-phase resolver returns the most-recent producer's value; decorators stack on the current producer and are NOT evicted.
- **Filters internally registered as `<name>Filter` providers.** `module.filter('currency', factory)` is functionally equivalent to `module.provider('currencyFilter', { $get: factory })` — `injector.get('currencyFilter') === $filter('currency')` (same reference). The `<name>Filter` naming makes decorators work for free: `module.decorator('currencyFilter', ['$delegate', …])` wraps the underlying filter and is visible through both lookup paths. The `$stateful = true` flag on a filter function (not the factory) opts the filter out of the digest's input-identity short-circuit; all nine built-ins are stateless. `$locale` is a single `module.factory('$locale', () => myLocale)` swap — `currency`/`number`/`date` read it lazily on each invocation, so config-time replacement takes immediate effect at run-time. Unknown filters surface as `Unknown filter: <name>` (a `FilterLookupError` instance) routed through `$exceptionHandler` with cause `'$filter'` at digest time; the digest continues. `parse(expr)` conservatively marks any filter-containing expression as `constant: false` at parse time; scope re-checks at watch install once `$filter` is reachable, upgrading to a constant-watch when the input/args are structurally constant AND no filter in the chain is `$stateful`.
- **Directive registration accumulates per name (no last-wins).** Two `$compileProvider.directive('foo', factoryA)` + `.directive('foo', factoryB)` calls produce TWO directives that both run on `<div foo>`. Mirrors AngularJS exactly. For services / filters / providers the rule is last-wins; for directives it is accumulation. Internally each name is a `<name>Directive` provider whose `$get` returns the array of normalized directive objects, so `injector.get('fooDirective')` returns both factories' resolved entries. Decorators on `<name>Directive` wrap the WHOLE array — the `$delegate` is the array, and the decorator may filter or wrap individual entries.
- **`.directive` / `.controller` / `.filter` module-DSL methods are pure config-block sugar (spec 021 / 016).** Each pushes ONE config block onto the module's `$$configBlocks` that forwards verbatim — `.directive` → `$compileProvider.directive`, `.controller` → `$controllerProvider.register`, `.filter` → `$filterProvider.register`. The DSL owns no registry and adds no validation: accumulation (`.directive`), last-wins (`.controller`), error messages, and registration timing are all inherited from the providers — tests verify the inheritance, they don't re-implement it. The `@di → @compiler` / `@di → @controller` references in `src/di/module.ts` are **`import type`-only** (erased at build), so `@di` keeps zero runtime dependency on `@compiler` / `@controller` and the module-boundary invariant holds — `.filter`'s `import type { … } from '@filter/...'` is the established precedent. **Registry-widening asymmetry:** single-name `.directive('x', …)` widens the typed `Registry` with `xDirective: Directive[]` (the `<name>Directive` provider is genuinely injector-resolvable, like `<name>Filter`), but bulk-map `.directive({…})` and BOTH `.controller` forms do NOT widen — controllers live in `$ControllerProvider`'s private `$$registry` Map and are never injector-resolvable, so there is no key to widen with. As with `.filter`, the runtime methods live on the `Module` class while the dependency-typed overloads live on the `TypedModule` interface — the same class-vs-interface split that avoids putting `Registry` in a contravariant position and breaking `Module` subtype variance.
- **Compile-phase mutation runs once per template.** Compile fns run during the walker's recursive descent and return link functions; the same compiled subtree can be linked against different scopes by re-invoking the linker, but compile runs exactly once. Directives that need template-time DOM mutation (adding classes, setting attributes that affect child compilation) put it in `compile`, not `link`. The `Attributes` instance is created during compile and shared across every link invocation — state stored on `attrs` (e.g. observers from `$observe`) accumulates across linker calls, so re-linking the same template against multiple scopes is supported but uncommon.
- **`$observe` lazily installs the per-attribute watch on first observer registration.** A directive that calls `attrs.$observe('href', fn)` triggers `$interpolate(attrs.href, true)` exactly once; if interpolated, a single `scope.$watch(interpolateFn, listener)` is installed and reused for all subsequent observers of the same attribute name. Static attributes get a one-shot `$evalAsync` notification per observer — no watch is wired. Without `$observe` registrations, no per-attribute watches exist at all, so an element with hundreds of static-looking attributes costs zero digest time.
- **`'$compile'` cause token is the 10th `EXCEPTION_HANDLER_CAUSES` entry.** Public-API additive change introduced in spec 017 (the `'$filter'` precedent from spec 016 is the prior pattern). Used for directive factory invocation (lazy at `<name>Directive` provider `$get`), compile, pre-link, post-link, and the `$set`-driven `$observe` callback notification path. The static-attribute `$observe` first-fire (scheduled via `$evalAsync`) routes through the standard `'$evalAsync'` cause — both surfaces are documented and tested.
- **Isolate scope intentionally created via `Scope.$new(true)` (the existing core API).** The compiler does NOT reimplement isolate scope — `parentScope.$new(true)` fires once per element bearing a `scope: { … }` directive; two such directives on the same element throw `MultipleIsolateScopeError` routed via `$exceptionHandler('$compile')` from the per-element linker. A `bindToController: {…}`-only directive does NOT create its own isolate scope (it consumes whatever scope the element already has), so it cannot trigger the multi-isolate conflict. `scope: false | true` from spec 017 still works unchanged; the per-element linker discriminates "child" vs "isolate" by checking `isolateBindings != null` on the normalized directive.
- **Raw `Element` argument is the deliberate choice.** No jqLite shim ships in spec 017; the link signature is `(scope, element, attrs)` where `element` is a native `Element` — or a `Comment` for an M-restricted match. The Phase 5 `angular.element` compat layer may layer on top without changing the link signature, so directives written today against raw `Element` will keep working when jqLite-style sugar lands.
- **Child-scope cleanup via private element properties.** When `scope: true` is requested, the compiler stashes the child scope on `(element as any).$$ngScope` and lets directives push cleanup callbacks onto `(element as any).$$ngCleanupQueue` — both non-enumerable. Future structural directives (`ng-if`, `ng-repeat`, …) MUST call `destroyElementScope(element)` before removing nodes; there is no `MutationObserver` or auto-detection. Without that call, removed-subtree scopes stay attached to the parent's watcher tree forever.
- **Transclusion scope is a child of the OUTER scope, not the directive's.** The scope the directive was linked against (`parentScope` inside the per-element linker) is captured by the `$transclude` closure BEFORE the `scope: true` `$new()` runs, so every clone's transclusion scope is `parentScope.$new()` — never `directiveScope.$new()`. This is what lets `<my-card scope="true">` keep its own variable namespace internally while consumer markup inside `<my-card>` still binds to OUTER variables.
- **Captured children compile EXACTLY ONCE at capture time.** Capture runs during the host directive's compile phase, before the walker descends into the host's child positions, so the OUTER walker sees no children to link. Each `$transclude(...)` call deep-clones the master via `Node.cloneNode(true)` and re-links the clone against a fresh transclusion scope through the internal `cloneMap` indirection (the public `Linker` type is unchanged). The master fragment is never inserted into the DOM.
- **Multi-clone is supported.** Each `$transclude(...)` call yields an independent clone with an independent scope, pushed onto the host element's `$$ngCleanupQueue` so `destroyElementScope(host)` tears every clone down regardless of how many were produced. This is the infrastructure `ng-repeat` will lean on. The outer scope's `$destroy()` also tears clones down via normal scope-tree propagation — both paths converge on `transcludedScope.$$destroyed === true`.
- **`ng-transclude` finds its enclosing transcluding directive by walking `element.parentElement`** until it finds an ancestor carrying the non-enumerable `$$ngBoundTransclude` slot. Decoupled from the walker — works for programmatically-inserted markers AND (in the future) for markers inside compiled `template` strings. The slot stashes `{ fn, declaredSlots, kind, directiveName }` so the marker validates slot-name lookups before invoking `$transclude`.
- **`transclude: 'element'` is deliberately deferred and REJECTED at registration** so a future structural-directives spec can add it without a silent semantic change. The `'$compile'` cause is reused for every transclusion error site — no new `EXCEPTION_HANDLER_CAUSES` entry (the tuple stays at 10). The nine new error classes are exported from `@compiler/index`; registration-time errors route via the existing factory try/catch in `$$buildDirectiveArrayProvider`, link-time errors route via direct `invokeExceptionHandler(handler, err, '$compile')` calls.
- **Template install slots AFTER transclude capture + BEFORE the per-directive compile loop on the host element.** The `compileElementOrComment` pre-pass detects the template-declaring directive AFTER spec 018's transclude capture has stashed `$$ngBoundTransclude` on the host, then replaces the host's children with the resolved template nodes BEFORE the matched-directive `compile` functions run. - **`$compile`'s public `Linker` signature is unchanged in spec 019.** `(scope: Scope) => Element | NodeList | Comment` is preserved even when `templateUrl` directives are present in the compiled subtree. Spec-017/018 callers remain assignable. Tests that observe post-install DOM `await Promise.resolve()` at least twice (defensively three times) to flush the drain chain.

- **`replace: true` is REJECTED permanently** at registration via `ReplaceTrueNotSupportedError`. AngularJS 1.x officially deprecated it; this project will NOT ship it. Templates always become the host element's children — the host element itself (tag, attributes, listeners) is preserved. `replace: false` and `replace: undefined` are accepted; any other value also throws `ReplaceTrueNotSupportedError`.
- **Function-form templates are called EXACTLY ONCE per compile invocation per host element.** The resolved string is memoized on the directive's local entry; subsequent `linker(scope1); linker(scope2)` invocations against the same compiled tree reuse the computed string. This applies equally to `template: (el, attrs) => string` and `templateUrl: (el, attrs) => string`. A function-form template that throws or returns a non-string value routes via `$exceptionHandler('$compile')` (`TemplateFunctionReturnedNonStringError` / `TemplateUrlFunctionReturnedNonStringError`) and leaves the host element empty.

- **Controllers register through `$controllerProvider.register(name, fn)`, instantiate through `$controller(nameOrFn, locals?, ident?)`.** The single alias parser `CONTROLLER_NAME_ALIAS_RE = /^(\S+?)(\s+as\s+([\w$]+))?\s*$/` and the shared `IDENT_RE = /^[A-Za-z_$][\w$]*$/` are exported from `@controller/controller.ts` so the directive-side `controllerAs` validation in `normalizeDirective` uses the same source of truth as `parseControllerName`; an explicit `ident` argument supersedes the `'Foo as vm'` alias suffix when both are present. `$controllerProvider.register` is config-phase only — the captured `$$getPhase` thunk on `ProvideService` (`$provide.$$getPhase`, an internal hook surfaced in spec 020 for cross-provider use — app code MUST NOT call it) trips the guard even on a `$controllerProvider` reference saved during config and called from a `run()` block; `$controller` itself is run-phase only, reachable via `injector.get('$controller')` after `createInjector` finishes. - **The four binding-wiring strategies are NOT symmetric.** `=` installs both a parent → local watch AND a reverse local → parent watch (the reverse path is suppressed for non-assignable parent expressions — non-assignable `=` silently degrades to one-way, deliberately NOT throwing `$compile:nonassign`); the last-digest-value reconciliation guards against ping-pong loops. `@` does NOT use `attrs.$observe` — it interpolates against the PARENT scope (not the isolate) via `interpolate(rawAttr, true)` and installs the watcher ON the isolate scope so `<my-dir title="{{outerName}}">` resolves `outerName` in the outer namespace; this is a deliberate divergence from a naive `attrs.$observe` reading, which would resolve against the isolate scope (where `outerName` does not exist). `<` synchronously seeds the local at link time before the watcher runs (so `$onInit` sees a populated value), then installs a single one-way watch with the first-fire `newValue === oldValue` sentinel suppressed in the `onChange` callback. `&` is a closure over `parentScope.$eval(expr, locals)` — no watcher; the target's slot value IS the callable.

- **`ng-repeat` derives default identities via a closure-local `WeakMap<object, string>` plus a monotonic counter — NO `$$hashKey` injection.** Deliberate AngularJS divergence (spec 028 Slice 2). Don't wire a lenient path behind it; there is no annotate-by-source-parse to gate.
- **`$rootElement` rides the additive optional `seed` seam in `createInjector` — headless passes nothing (spec 036).** `createInjector(modules, { seed?: Record<string, unknown> })` writes each seed entry directly into `providerCache` — the SAME slot the injector self-seeds `$injector` into — so a seeded name resolves through the cache-hit fast path in `get` exactly like `$injector`. The DOM `bootstrap` seeds `{ $rootElement: element }` so the started element is injectable; `bootstrapInjector` passes no options, so the loop is empty and behavior is byte-identical to self-seeding only `$injector`. The seam is generic (any name/value), not `$rootElement`-specific — future page-level services (`$document`, `$location`) can reuse it without touching the recipe maps.
- **The double-bootstrap guard reads the `$$ngBootstrapped` MARKER, not an attached `$injector` (spec 036).** `bootstrap` stamps a single non-enumerable `$$ngBootstrapped: true` slot (`src/bootstrap/element-marker.ts`, mirroring the compiler's `cleanup.ts` element-slot pattern) the first time it starts an app on an element; a second `bootstrap(...)` on the same element reads that marker via the `in`-operator guard and throws `AlreadyBootstrappedError`. Injector attachment (`$$ngInjector`) is a SEPARATE, opt-in (`config.attachToElement`) convenience that the guard NEVER reads — so leaving attachment off (the default) keeps the element clean of the injector reference while the guard still fires. An `ng-app` nested inside an already-started region hits this same guard through `autoBootstrap → bootstrap` and throws (intended, not suppressed).
- **All bootstrap failures throw SYNCHRONOUSLY to the caller — never `$exceptionHandler` (spec 036).** `BootstrapTargetMissingError` (null/undefined target), `AlreadyBootstrappedError` (re-bootstrap), and `getModule`'s reused `Module not found: <name>` are programmer errors surfaced AT THE CALL SITE, not routed through `$exceptionHandler`. The two error classes (`src/bootstrap/bootstrap-error.ts`) carry literal `name` brands for `instanceof` narrowing and stable test-locked messages. `EXCEPTION_HANDLER_CAUSES.length` STILL 10 after spec 036 — no new cause token.
- **`$rootScope` is a LAZY singleton factory on `ngModule` (spec 036 Slice 2).** Registered via `.factory('$rootScope', [() => Scope.create()])` in `src/core/ng-module.ts` (array-wrapped to satisfy strict `annotate`): `Scope.create()` runs only on the FIRST `injector.get('$rootScope')`, and the DI cache returns the SAME reference on every subsequent `get` — so the `bootstrap` result's `rootScope === injector.get('$rootScope')`. The `ng` typed registry was widened with `$rootScope: Scope`. A documented `TODO(future)` will switch construction to `Scope.create({ filterLookup, exceptionHandler })` once that doesn't force `@core`'s `ngModule` to pull `$filter` into the construction path.

## Coding conventions

- **TypeScript strict** (`strict: true` + `noUncheckedIndexedAccess`). No `any` — the single existing cast in `src/core/utils.ts:229` is the ceiling, not a precedent.
- **Every `eslint-disable` comment must carry an inline justification** (`-- reason`). CI enforces lint.
- **No explicit return types** when TS inference handles them — let inference do the work. Annotate only on exported public-API boundaries where the declared shape is part of the contract.
- **Imports**: use path aliases (`@core`, `@parser`, `@di`, `@compiler`). `no-restricted-imports` blocks `../*` relative climbing.
- **File naming**: kebab-case (`scope-watch-delegates.ts`, `ast-flags.ts`). Tests under `src/<module>/__tests__/*.test.ts`.
- **File size target**: under 500 lines per source file. Refactor candidates today: `src/core/scope.ts` (827), `src/di/module.ts` (776), `src/di/injector.ts` (734).
