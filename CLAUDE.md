# my-own-angularjs ŌĆö AI context

A from-scratch TypeScript reimplementation of AngularJS. Goal: **clarity over performance**. Every feature goes through the AWOS spec-driven workflow before code.

## Commands

```bash
pnpm test          # vitest (jsdom env, 90% line-coverage threshold)
pnpm typecheck     # tsc --noEmit, strict + noUncheckedIndexedAccess
pnpm lint          # ESLint strictTypeChecked preset
pnpm format        # Prettier (semi, singleQuote, printWidth 120)
pnpm build         # Rollup dual ESM+CJS with .d.ts for each subpath
```

CI (`.github/workflows/ci.yml`) gates on: lint ŌåÆ format:check ŌåÆ typecheck ŌåÆ test. `build` is not yet in CI.

## Non-obvious invariants

- **No `new Function()` / no `eval()`** ŌĆö the expression parser uses a **tree-walking interpreter** deliberately. This avoids CSP violations and is part of the project's security posture. Don't "optimize" by generating code strings.
- **Digest TTL contract** ŌĆö configurable via `Scope.create({ ttl: 20 })` (default 10). TTL breach throws with the watch function source in the error to help identify unstable watchers.
- **One-time bindings & constant watches** (spec 010) ŌĆö the parser attaches `literal` / `constant` / `oneTime` flags on AST nodes. Scope wires `oneTimeWatchDelegate` (literal) or `oneTimeLiteralWatchDelegate` for `::expr` expressions; constant expressions use `constantWatchDelegate`. When modifying the watcher wiring, preserve those delegate selections.
- **Module boundary rule** ŌĆö `parser/*` and `di/*` depend only on `@core` (prefer importing from `@core/index`, not `@core/utils` directly). `core/scope.ts` intentionally depends on `@parser/index` because scopes evaluate expression strings. `compiler/` has no dependencies yet.
- **Error handling in digest** ŌĆö listener/watch exceptions are logged with `console.error('...', e)` and the digest continues. Don't swallow silently and don't abort the digest loop on a single listener failure.
- **Strict mode is frozen after the config phase** ŌĆö `$sceProvider.enabled(false)` is the only way to disable SCE. Once the injector finishes config, `$sce.isEnabled()` is permanent. Strict-OFF turns both `trustAs*` and `getTrusted*` into total pass-throughs (no wrapper classes are constructed).
- **Trusted values are per-context nominal classes** ŌĆö `TrustedResourceUrl extends TrustedUrl`, so a trusted resource URL is accepted where a trusted URL is expected (not vice-versa). Identity is checked via `instanceof`, not a string-based brand. Do NOT "optimize" to a single branded wrapper ŌĆö the subtype rule matters for AngularJS parity.
- **`ngSanitize` is opt-in, never registered on the core `ng` module** ŌĆö apps must list `'ngSanitize'` in their dependency chain. When loaded, `$sce.getTrustedHtml(plainString)` automatically routes through `$sanitize` via a lazy `$injector.has('$sanitize')` lookup in `$SceProvider.$get` ŌĆö no hard dependency from `$sce` to `ngSanitize`, no decoration. Removing the `$injector` dep from `$SceProvider.$get` would silently break this integration; the regression test in `src/sanitize/__tests__/sanitize-sce.test.ts` is the guard.
- **The digest's "log and continue" contract is preserved through `$exceptionHandler`.** A failing watcher / listener / async task is reported via the configured handler and the digest proceeds; only TTL exhaustion re-throws (after first reporting via the handler, cause `'$digest'`). The default handler is `console.error`, so today's logs continue to appear unchanged. A custom handler that itself throws is caught by `invokeExceptionHandler` and degrades to `console.error` ŌĆö the digest still does not crash. The nine `EXCEPTION_HANDLER_CAUSES` tokens are part of the public contract; future specs that add framework-internal call sites must extend the list as a public-API change.
- **`$provide` is config-phase only** ŌĆö the six methods (`factory`, `service`, `value`, `constant`, `provider`, `decorator`) throw synchronously with `$provide.<method> is only callable during the config phase; calling it after the run phase begins is not supported` whenever they're invoked outside a `config()` block, including via a `$provide` reference captured during config and called later. This out-of-phase error is a programming error and is **not** routed through `$exceptionHandler` ŌĆö it surfaces directly to the caller. Constants are protected by an override guard: `value` / `factory` / `service` / `provider` / `decorator` (whether through the module DSL or `$provide`) targeting a name already registered as a `.constant` throw `Cannot override constant "<name>" ŌĆö already registered via .constant(...)`. Within the unified registration timeline, a new producer recipe wipes prior producer entries for the same name from the other backing maps so the run-phase resolver returns the most-recent producer's value; decorators stack on the current producer and are NOT evicted.
- **Filters internally registered as `<name>Filter` providers.** `module.filter('currency', factory)` is functionally equivalent to `module.provider('currencyFilter', { $get: factory })` ŌĆö `injector.get('currencyFilter') === $filter('currency')` (same reference). The `<name>Filter` naming makes decorators work for free: `module.decorator('currencyFilter', ['$delegate', ŌĆ”])` wraps the underlying filter and is visible through both lookup paths. The `$stateful = true` flag on a filter function (not the factory) opts the filter out of the digest's input-identity short-circuit; all nine built-ins are stateless. `$locale` is a single `module.factory('$locale', () => myLocale)` swap ŌĆö `currency`/`number`/`date` read it lazily on each invocation, so config-time replacement takes immediate effect at run-time. Unknown filters surface as `Unknown filter: <name>` (a `FilterLookupError` instance) routed through `$exceptionHandler` with cause `'$filter'` at digest time; the digest continues. `parse(expr)` conservatively marks any filter-containing expression as `constant: false` at parse time; scope re-checks at watch install once `$filter` is reachable, upgrading to a constant-watch when the input/args are structurally constant AND no filter in the chain is `$stateful`.
- **Directive registration accumulates per name (no last-wins).** Two `$compileProvider.directive('foo', factoryA)` + `.directive('foo', factoryB)` calls produce TWO directives that both run on `<div foo>`. Mirrors AngularJS exactly. For services / filters / providers the rule is last-wins; for directives it is accumulation. Internally each name is a `<name>Directive` provider whose `$get` returns the array of normalized directive objects, so `injector.get('fooDirective')` returns both factories' resolved entries. Decorators on `<name>Directive` wrap the WHOLE array ŌĆö the `$delegate` is the array, and the decorator may filter or wrap individual entries.
- **`.directive` / `.controller` / `.filter` module-DSL methods are pure config-block sugar (spec 021 / 016).** Each pushes ONE config block onto the module's `$$configBlocks` that forwards verbatim ŌĆö `.directive` ŌåÆ `$compileProvider.directive`, `.controller` ŌåÆ `$controllerProvider.register`, `.filter` ŌåÆ `$filterProvider.register`. The DSL owns no registry and adds no validation: accumulation (`.directive`), last-wins (`.controller`), error messages, and registration timing are all inherited from the providers ŌĆö tests verify the inheritance, they don't re-implement it. The `@di ŌåÆ @compiler` / `@di ŌåÆ @controller` references in `src/di/module.ts` are **`import type`-only** (erased at build), so `@di` keeps zero runtime dependency on `@compiler` / `@controller` and the module-boundary invariant holds ŌĆö `.filter`'s `import type { ŌĆ” } from '@filter/...'` is the established precedent. **Registry-widening asymmetry:** single-name `.directive('x', ŌĆ”)` widens the typed `Registry` with `xDirective: Directive[]` (the `<name>Directive` provider is genuinely injector-resolvable, like `<name>Filter`), but bulk-map `.directive({ŌĆ”})` and BOTH `.controller` forms do NOT widen ŌĆö controllers live in `$ControllerProvider`'s private `$$registry` Map and are never injector-resolvable, so there is no key to widen with. As with `.filter`, the runtime methods live on the `Module` class while the dependency-typed overloads live on the `TypedModule` interface ŌĆö the same class-vs-interface split that avoids putting `Registry` in a contravariant position and breaking `Module` subtype variance.
- **Compile-phase mutation runs once per template.** Compile fns run during the walker's recursive descent and return link functions; the same compiled subtree can be linked against different scopes by re-invoking the linker, but compile runs exactly once. Directives that need template-time DOM mutation (adding classes, setting attributes that affect child compilation) put it in `compile`, not `link`. The `Attributes` instance is created during compile and shared across every link invocation ŌĆö state stored on `attrs` (e.g. observers from `$observe`) accumulates across linker calls, so re-linking the same template against multiple scopes is supported but uncommon.
- **`$observe` lazily installs the per-attribute watch on first observer registration.** A directive that calls `attrs.$observe('href', fn)` triggers `$interpolate(attrs.href, true)` exactly once; if interpolated, a single `scope.$watch(interpolateFn, listener)` is installed and reused for all subsequent observers of the same attribute name. Static attributes get a one-shot `$evalAsync` notification per observer ŌĆö no watch is wired. Without `$observe` registrations, no per-attribute watches exist at all, so an element with hundreds of static-looking attributes costs zero digest time.
- **`'$compile'` cause token is the 10th `EXCEPTION_HANDLER_CAUSES` entry.** Public-API additive change introduced in spec 017 (the `'$filter'` precedent from spec 016 is the prior pattern). Used for directive factory invocation (lazy at `<name>Directive` provider `$get`), compile, pre-link, post-link, and the `$set`-driven `$observe` callback notification path. The static-attribute `$observe` first-fire (scheduled via `$evalAsync`) routes through the standard `'$evalAsync'` cause ŌĆö both surfaces are documented and tested.
- **Isolate scope intentionally created via `Scope.$new(true)` (the existing core API).** The compiler does NOT reimplement isolate scope ŌĆö `parentScope.$new(true)` fires once per element bearing a `scope: { ŌĆ” }` directive; two such directives on the same element throw `MultipleIsolateScopeError` routed via `$exceptionHandler('$compile')` from the per-element linker. A `bindToController: {ŌĆ”}`-only directive does NOT create its own isolate scope (it consumes whatever scope the element already has), so it cannot trigger the multi-isolate conflict. `scope: false | true` from spec 017 still works unchanged; the per-element linker discriminates "child" vs "isolate" by checking `isolateBindings != null` on the normalized directive.
- **Raw `Element` argument is the deliberate choice.** No jqLite shim ships in spec 017; the link signature is `(scope, element, attrs)` where `element` is a native `Element` ŌĆö or a `Comment` for an M-restricted match. The Phase 5 `angular.element` compat layer may layer on top without changing the link signature, so directives written today against raw `Element` will keep working when jqLite-style sugar lands.
- **Child-scope cleanup via private element properties.** When `scope: true` is requested, the compiler stashes the child scope on `(element as any).$$ngScope` and lets directives push cleanup callbacks onto `(element as any).$$ngCleanupQueue` ŌĆö both non-enumerable. Future structural directives (`ng-if`, `ng-repeat`, ŌĆ”) MUST call `destroyElementScope(element)` before removing nodes; there is no `MutationObserver` or auto-detection. Without that call, removed-subtree scopes stay attached to the parent's watcher tree forever.
- **Transclusion scope is a child of the OUTER scope, not the directive's.** The scope the directive was linked against (`parentScope` inside the per-element linker) is captured by the `$transclude` closure BEFORE the `scope: true` `$new()` runs, so every clone's transclusion scope is `parentScope.$new()` ŌĆö never `directiveScope.$new()`. This is what lets `<my-card scope="true">` keep its own variable namespace internally while consumer markup inside `<my-card>` still binds to OUTER variables.
- **Captured children compile EXACTLY ONCE at capture time.** Capture runs during the host directive's compile phase, before the walker descends into the host's child positions, so the OUTER walker sees no children to link. Each `$transclude(...)` call deep-clones the master via `Node.cloneNode(true)` and re-links the clone against a fresh transclusion scope through the internal `cloneMap` indirection (the public `Linker` type is unchanged). The master fragment is never inserted into the DOM.
- **Multi-clone is supported.** Each `$transclude(...)` call yields an independent clone with an independent scope, pushed onto the host element's `$$ngCleanupQueue` so `destroyElementScope(host)` tears every clone down regardless of how many were produced. This is the infrastructure `ng-repeat` will lean on. The outer scope's `$destroy()` also tears clones down via normal scope-tree propagation ŌĆö both paths converge on `transcludedScope.$$destroyed === true`.
- **`ng-transclude` finds its enclosing transcluding directive by walking `element.parentElement`** until it finds an ancestor carrying the non-enumerable `$$ngBoundTransclude` slot. Decoupled from the walker ŌĆö works for programmatically-inserted markers AND (in the future) for markers inside compiled `template` strings. The slot stashes `{ fn, declaredSlots, kind, directiveName }` so the marker validates slot-name lookups before invoking `$transclude`.
- **`transclude: 'element'` is deliberately deferred and REJECTED at registration** so a future structural-directives spec can add it without a silent semantic change. The `'$compile'` cause is reused for every transclusion error site ŌĆö no new `EXCEPTION_HANDLER_CAUSES` entry (the tuple stays at 10). The nine new error classes are exported from `@compiler/index`; registration-time errors route via the existing factory try/catch in `$$buildDirectiveArrayProvider`, link-time errors route via direct `invokeExceptionHandler(handler, err, '$compile')` calls.
- **Template install slots AFTER transclude capture + BEFORE the per-directive compile loop on the host element.** The `compileElementOrComment` pre-pass detects the template-declaring directive AFTER spec 018's transclude capture has stashed `$$ngBoundTransclude` on the host, then replaces the host's children with the resolved template nodes BEFORE the matched-directive `compile` functions run. As a result the canonical `transclude: true` + `template` + `<ng-transclude>` wrapper pattern works without extra wiring ŌĆö the consumer's children are already captured when the template overwrites them, and `<ng-transclude>` markers inside the template find the host's `$$ngBoundTransclude` via the existing `parentElement` walk.
- **`$compile`'s public `Linker` signature is unchanged in spec 019.** `(scope: Scope) => Element | NodeList | Comment` is preserved even when `templateUrl` directives are present in the compiled subtree. Async resolution happens internally ŌĆö the top-level `$compile` schedules `Promise.resolve().then(drainDeferredTemplateQueue)` after the synchronous walker returns. Spec-017/018 callers remain assignable. Tests that observe post-install DOM `await Promise.resolve()` at least twice (defensively three times) to flush the drain chain.
- **`$templateRequest` deduplicates concurrent fetches via an `inFlight: Map<string, Promise<string>>`.** Two requests for the same URL before the first resolves share a single network call (and the cache-write side effect on resolution). Cleanup happens after settlement so a subsequent request after `$templateCache.remove(url)` re-fetches. Tests that exercise the deduplication seam usually inject a deferred mock fetcher via `createTemplateRequest({ cache, fetcher })`.
- **`replace: true` is REJECTED permanently** at registration via `ReplaceTrueNotSupportedError`. AngularJS 1.x officially deprecated it; this project will NOT ship it. Templates always become the host element's children ŌĆö the host element itself (tag, attributes, listeners) is preserved. `replace: false` and `replace: undefined` are accepted; any other value also throws `ReplaceTrueNotSupportedError`.
- **Function-form templates are called EXACTLY ONCE per compile invocation per host element.** The resolved string is memoized on the directive's local entry; subsequent `linker(scope1); linker(scope2)` invocations against the same compiled tree reuse the computed string. This applies equally to `template: (el, attrs) => string` and `templateUrl: (el, attrs) => string`. A function-form template that throws or returns a non-string value routes via `$exceptionHandler('$compile')` (`TemplateFunctionReturnedNonStringError` / `TemplateUrlFunctionReturnedNonStringError`) and leaves the host element empty.
- **`'$compile'` cause token is reused for every template-related error site.** The `EXCEPTION_HANDLER_CAUSES` tuple stays at 10 entries; spec 019 introduces NO new cause token. Template-fetch failures, multi-template-on-element, function-form throws / non-string returns, registration-phase validation errors (`InvalidTemplateValueError`, `InvalidTemplateUrlValueError`, `EmptyTemplateError`, `EmptyTemplateUrlError`, `TemplateAndTemplateUrlCombinedError`, `ReplaceTrueNotSupportedError`), and link-phase multi-template detection (`MultipleTemplateDirectivesError`) all route through the same handler the rest of `$compile` uses. A custom handler that itself throws is caught by spec-014's `invokeExceptionHandler` recursion guard and falls back to `console.error` ŌĆö template loading does NOT crash.
- **Controllers register through `$controllerProvider.register(name, fn)`, instantiate through `$controller(nameOrFn, locals?, ident?)`.** The single alias parser `CONTROLLER_NAME_ALIAS_RE = /^(\S+?)(\s+as\s+([\w$]+))?\s*$/` and the shared `IDENT_RE = /^[A-Za-z_$][\w$]*$/` are exported from `@controller/controller.ts` so the directive-side `controllerAs` validation in `normalizeDirective` uses the same source of truth as `parseControllerName`; an explicit `ident` argument supersedes the `'Foo as vm'` alias suffix when both are present. `$controllerProvider.register` is config-phase only ŌĆö the captured `$$getPhase` thunk on `ProvideService` (`$provide.$$getPhase`, an internal hook surfaced in spec 020 for cross-provider use ŌĆö app code MUST NOT call it) trips the guard even on a `$controllerProvider` reference saved during config and called from a `run()` block; `$controller` itself is run-phase only, reachable via `injector.get('$controller')` after `createInjector` finishes. The compiler's per-element controller seam runs ONCE per directive declaring `controller`, AFTER `$transclude` setup and BEFORE the directive's own pre-link (the shared `runControllerSeam` helper closes over `$controller` + `exceptionHandler` and fires from BOTH the inline-link path and the post-template-install path inside `buildPostTemplateLinker`, so `templateUrl` directives still see their controller at the same point in the post-template link cycle); throws route via the existing `'$compile'` cause ŌĆö no new `EXCEPTION_HANDLER_CAUSES` entry, the tuple stays at 10. `controllerAs`-without-`controller` is REJECTED AT DIRECTIVE REGISTRATION (in `normalizeDirective`'s `normalizeController` block, routed via the existing factory `try/catch` in `$$buildDirectiveArrayProvider` through `$exceptionHandler('$compile')`), NOT at link time. User data is byte-identical before and after `ng-repeat` consumes it ŌĆö a consumer feeding the same array to a serializer or fetch body builder sees no mystery `$$hashKey` property. `Object.freeze`d items work transparently (AngularJS's `$$hashKey` injection throws on frozen first-encounter; the WeakMap stores the mapping externally). Identity is by reference, not by value ŌĆö mutating an item in place keeps its identity; authors who want structural identity supply `track by todo.id`. For object collections (`(key, value) in obj`) the identity formula folds the property key in too ŌĆö `'key:${objKey}|${valueIdentity}'` ŌĆö so the same value under two distinct keys (`{a: 1, b: 1}`) does NOT falsely collapse to a duplicate-key throw. A fresh tracker is created per directive instance (per link invocation); two `ng-repeat` instances over the same collection produce independent identity namespaces. The hook lives in `src/compiler/ng-repeat-identity.ts` ŌĆö `createIdentityTracker()` returns `{ getIdentity(value): string }`.
- **`ng-repeat`'s `as ALIAS` write happens BEFORE row reconciliation in the same listener fire.** `publishAlias(rawCollection, normalized)` is the FIRST observable side effect of `reconcile` ŌĆö runs before the non-iterable bail, before identity computation, before the diff walk. Sibling markup later in the digest tree (`<p ng-if="!visible.length">No matches.</p>`) therefore sees the new value in the same turn ŌĆö no extra digest needed for the canonical empty-state pattern. Per-shape value contract: array iteration ŌåÆ raw post-filter array; object iteration ŌåÆ normalized `[{ key, value }]` array (sibling markup reads `.length` uniformly); non-iterable ŌåÆ `[]` (so `!visible.length` fires uniformly across "no matches", "not loaded yet", "destroyed"). A duplicate-key throw AFTER publication does NOT roll the alias back ŌĆö the alias is the input surface, rows are the reconciliation surface, independent outputs. The parser-side `NgRepeatBadAliasError` (Slice 1) prevents the alias from colliding with the item / key / value names in the same expression and with the six reserved per-row locals (`$index`, `$first`, `$last`, `$middle`, `$even`, `$odd`); generic shadowing of other parent-scope properties is documented as the author's responsibility, matching AngularJS.
- **`ng-repeat`'s duplicate-key detection routes via `$exceptionHandler('$compile')`, NOT `'watchListener'`.** The directive wraps `reconcile(newCollection)` in its OWN `try/catch` inside the `$watchCollection` listener: a `NgRepeatDuplicateKeyError` throw is caught locally, `tearDownAllRows()` clears any partial state, and `invokeExceptionHandler($exceptionHandler, err, '$compile')` routes via the `'$compile'` cause token. The watcher returns cleanly, so the digest's standard `'watchListener'` catch path is NOT exercised on the duplicate-key path ŌĆö apps that branch their `$exceptionHandler` on cause must filter on `'$compile'` for this signal. `EXCEPTION_HANDLER_CAUSES.length` STILL at 10 after spec 028 ŌĆö all four new error classes (`NgRepeatBadIteratorExpressionError`, `NgRepeatBadIdentifierError`, `NgRepeatBadAliasError`, `NgRepeatDuplicateKeyError`) reuse the existing `'$compile'` cause token; no new entry to the tuple.
- **`ng-repeat`'s `priority: 1000` wins same-element conflicts against the other structural directives** (`ng-if` at 600, `ng-include` at 400). So `<li ng-repeat="ŌĆ”" ng-if="ŌĆ”">` runs `ng-repeat` first and the spec 027 known gap surfaces (the spec-017 same-element terminal cutoff in `directive-collector.ts` silently drops the lower-priority `ng-if` instead of surfacing `MultipleTranscludeDirectivesError`). The canonical fix is nesting (`<li ng-repeat="ŌĆ”"><span ng-if="ŌĆ”">ŌĆ”</span></li>`); the same future spec slice that re-orders the multi-structural detection in spec 027 will fix this case too. Non-structural directives on the same host (`<li ng-repeat="ŌĆ”" ng-class="ŌĆ”">`) are unaffected ŌĆö `ng-class` is not `transclude: 'element'` and does not trip the cutoff.
- **`ng-repeat` row reuse on identity match preserves DOM-node identity AND the state inside.** On each `$watchCollection` fire the diff walk consults `currentRows`: an identity hit takes the REUSE branch ŌĆö scope, watchers, listeners, and DOM subtree are kept intact; only the six per-row locals + item/key bindings are updated; the `cloneRoot` is moved via `parentNode.insertBefore(cloneRoot, anchor.nextSibling)` so the DOM node itself is the same reference before and after the reorder. Focused `<input>` elements keep their selection range, form values persist, scroll positions inside the row survive. Only a duplicate-key throw OR a missing identity (item disappeared / freshly arrived) triggers fresh `$transclude(...)` / `$destroy()` cycles. The FRESH BUILD branch always populates locals + item/key bindings BEFORE DOM insertion so first-render watchers fire with correct values.
- **`ng-repeat` object iteration visits keys in alphabetical-string order (AngularJS-canonical).** `normalizeCollection` triages `Array.isArray(coll)` ŌåÆ array branch first, then `coll !== null && typeof coll === 'object'` ŌåÆ object branch with keys taken via `Object.keys(coll).sort()` (default string-comparison ordering). So `{'10': 'a', '2': 'b', '1': 'c'}` renders rows in the order `1 ŌåÆ c`, `10 ŌåÆ a`, `2 ŌåÆ b` (NOT numeric-sort order). Functions deliberately take the non-iterable bail (`typeof 'function'` is disjoint from `typeof 'object'`); the `!== null` check precedes `typeof === 'object'` because `typeof null === 'object'` in JavaScript. The object-collection identity-with-key formula (`'key:${objKey}|${valueIdentity}'`) means a property-value change is observable as a tear-down + fresh-build pair on the next digest, NOT an in-place update ŌĆö documented behavior, not a bug. `$watchCollection` shallow-watches object own keys, so in-place mutations (`scope.bag.bob = 2`, `delete scope.bag.bob`) fire the watch listener ŌĆö no reassignment needed for object collections.

## Coding conventions

- **TypeScript strict** (`strict: true` + `noUncheckedIndexedAccess`). No `any` ŌĆö the single existing cast in `src/core/utils.ts:229` is the ceiling, not a precedent.
- **Every `eslint-disable` comment must carry an inline justification** (`-- reason`). CI enforces lint.
- **No explicit return types** when TS inference handles them ŌĆö let inference do the work. Annotate only on exported public-API boundaries where the declared shape is part of the contract.
- **Imports**: use path aliases (`@core`, `@parser`, `@di`, `@compiler`). `no-restricted-imports` blocks `../*` relative climbing.
- **File naming**: kebab-case (`scope-watch-delegates.ts`, `ast-flags.ts`). Tests under `src/<module>/__tests__/*.test.ts`.
